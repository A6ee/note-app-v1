原本的問題：語音辨識被重複初始化，事件處理器互相覆蓋
更新版本：只保留單一初始化入口 initRecognition，且 recognition 已存在就直接 return
修改方式：刪除檔案中第二段獨立 recognition 建立與 onresult/onend 綁定，統一收斂到同一套 lifecycle
問題原因：重複初始化會造成 onresult/onend 行為不一致，容易出現 transcript 被覆蓋或重啟邏輯混亂

原本的問題：缺少完整辨識錯誤處理，失敗時使用者不知道發生什麼

修改方式：新增 getRecognitionErrorMessage，onerror 依錯誤碼分流；權限/裝置錯誤會中止流程並提示
問題原因：Web Speech API 常見錯誤若不分類，會變成 silent failure，使用者只看到「沒反應」

原本的問題：逐字稿暫存時機分散，存在覆寫/遺失風險
更新版本：統一用 getTranscriptText + persistTempTranscript 管理 temp_transcript
修改方式：onresult、cancel、stop、beforeunload 都走同一暫存 helper
問題原因：多處直接 setItem 容易寫法不一致，導致保存內容不完整或時機錯誤

原本的問題：stop/cancel/onend 互相干擾，存在競態

原本的問題： callGemini 的 AbortError 被當作可重試的網路錯誤，導致 25s 超時後繼續遞迴，最長可卡住 125 秒

原本的問題： 503 使用與 429 相同的 MAX_RETRY = 5，等待時間累計超過 31 秒

部署前仍需手動設定（原因與步驟）
目前程式端已完成大部分資安實作（包含 API 驗證、CSP 標頭設定、Firestore 規則檔），
但仍有幾項必須在雲端平台手動完成，否則會出現「程式已改、正式站未生效」的落差。

1. Firebase Rules 發布（必要）
原因：
本地檔案 firestore.rules 已更新，不代表雲端 Firestore 已生效；必須在 Firebase Console 發布。

步驟：
打開 Firebase Console → Firestore Database → Rules。
將 firestore.rules 內容貼上（或用 CLI 部署）。
按 Publish。
以非本人 UID 測試讀寫，確認被拒絕。

2. Vercel 環境變數設定（必要）
原因：
正式部署不會自動讀你電腦本地的 .env，API 在雲端執行時只會
讀 Vercel Project 的 Environment Variables。

步驟：
Vercel 專案 → Settings → Environment Variables。
新增：
GEMINI_API_KEY
UPSTASH_REDIS_REST_URL
UPSTASH_REDIS_REST_TOKEN
SECURITY_ALERT_WEBHOOK_URL（選填，做告警用）
變數至少套用到 Production（建議 Preview、Development 也一併設定）。
重新部署一次，讓新變數生效。

3. Upstash Redis 參數取得（P2 限流必要）
原因：
gemini.js 的平台層持久化限流依賴 Upstash Redis REST；
未設定會退回記憶體限流（在 serverless 下不穩定）。

步驟：
到 Upstash 建立 Redis Database。
在該 Database 的 REST API 區塊取得：
UPSTASH_REDIS_REST_URL
UPSTASH_REDIS_REST_TOKEN
回 Vercel 填入上述環境變數。

4. 安全標頭生效驗證（必要）
原因：
vercel.json 只是設定檔，仍需在正式網域確認回應標頭實際存在。

步驟：
部署後打開正式站。
DevTools → Network → 重新整理。
點第一個 document 請求（不是 css/js）。
檢查 Response Headers 是否包含：
Content-Security-Policy
X-Frame-Options
Referrer-Policy
Strict-Transport-Security

5. WAF / Bot 防護（建議，P2）
原因：
應用程式內限流只能擋一部分濫用，平台層 WAF 與 Bot 規則才能補齊大流量與惡意流量防護。

步驟：
Vercel Dashboard 啟用 WAF managed rules。
對 /api/* 啟用 Bot Protection 或 Challenge。
設定流量異常告警（含成本異常告警）。
